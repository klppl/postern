Sending email from your application

Replace pn_YOUR_KEY_HERE in every snippet below with the raw API key. The raw key is shown once when you create or rotate a key on the API keys page — copy it then. Recipients and the From address are configured per-key, not per-request.

Endpoint

POST {{.BaseURL}}/api/v1/send
Authorization: Bearer pn_YOUR_KEY_HERE
Content-Type: application/json

Request body

Two shapes are accepted. Inline content:

{
  "subject": "Welcome",
  "body": "Hi there.",
  "body_html": "<p>Hi there.</p>"
}

Or a template reference:

{
  "template_name": "welcome",
  "variables": { "name": "Alex" }
}

With a template, the subject and bodies come from the stored template and variables are substituted via {{"{{variable}}"}} Handlebars syntax. Pass "subject": "..." alongside a template to override the subject for one send.

Recipients

By default each API key carries its own fixed to, cc, and bcc list — those are used for every send and the request body cannot change them. This is the safest default: a leaked key cannot be used to email arbitrary addresses.

If a key has "Allow the API request to override recipients" enabled, you can pass to, cc, and/or bcc in the body to send to different addresses for that one call: 

{
  "subject": "Your invoice",
  "body": "See attached.",
  "to":  ["customer@example.com"],
  "cc":  ["billing@example.com"],
  "bcc": []
}
  • If you supply any of to/cc/bcc, you take over all three for that send — there is no field-level merge.
  • If you supply none, the key's defaults are used.
  • Total addresses across the three fields is capped at 50.
  • If the key does not have the override enabled, requests carrying recipients are rejected with 400 invalid_recipients.

Responses

StatusMeaningBody
202Queued{"message_id":"…","status":"queued"}
400Invalid JSON or missing subject/body{"error":"invalid_json","message":"…"}
401Missing or invalid Authorization{"error":"missing or malformed Authorization header"}
422Key has no recipients configured{"error":"no_recipients", …}
429Rate limit hit. Honor Retry-After.{"error":"rate_limited", …}

202 means accepted into the queue, not delivered. Watch Messages for the actual outcome.

curl

curl -X POST {{.BaseURL}}/api/v1/send \
  -H "Authorization: Bearer pn_YOUR_KEY_HERE" \
  -H "Content-Type: application/json" \
  -d '{"subject":"Welcome","body":"Hi there."}'

Go (stdlib)

package mailer

import (
    "bytes"
    "encoding/json"
    "fmt"
    "net/http"
)

const endpoint = "{{.BaseURL}}/api/v1/send"

func Send(apiKey string, payload map[string]any) error {
    body, _ := json.Marshal(payload)
    req, _ := http.NewRequest("POST", endpoint, bytes.NewReader(body))
    req.Header.Set("Authorization", "Bearer "+apiKey) // pn_YOUR_KEY_HERE
    req.Header.Set("Content-Type", "application/json")
    resp, err := http.DefaultClient.Do(req)
    if err != nil {
        return err
    }
    defer resp.Body.Close()
    if resp.StatusCode != http.StatusAccepted {
        return fmt.Errorf("postern: %s", resp.Status)
    }
    return nil
}

Python (requests)

import os, requests

requests.post(
    "{{.BaseURL}}/api/v1/send",
    headers={
        "Authorization": f"Bearer {os.environ['POSTERN_API_KEY']}",  # pn_YOUR_KEY_HERE
        "Content-Type": "application/json",
    },
    json={"subject": "Welcome", "body": "Hi there."},
).raise_for_status()

Operational tips

  • Read the API key from an environment variable or secret manager — never hardcode it.
  • Treat 429 as backoff signal; the Retry-After header tells you how long to wait.
  • Treat 5xx as transient; retry with your own backoff. Postern is queue-backed so duplicate enqueue from your client is the main risk — make your handler idempotent or de-dupe by your own message key.
  • The message_id returned in the 202 body is queryable on the Messages page.
